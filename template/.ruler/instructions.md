## P0 — SAFETY & PERMISSIONS (NON-NEGOTIABLE)

P0 overrides all other rules. If a P0 conflict exists with any skill, subagent, or convention, P0 wins.

### P0.1 Hard safety gates

- **`main` is off-limits.** MUST NEVER commit, push, force-push, merge, or rebase to `main`/`master`. MUST use a feature branch and a PR. The `permissions.deny` block in `.claude/settings.json` denies common patterns; treat the rule as absolute even when a pattern slips through.
- **Git/GitHub writes need explicit user approval** (commit, push, branch create, PR, merge, rebase, force, tag). Reads are free.
- **Deploy / publish need explicit user approval** (`vercel deploy`, `netlify deploy`, `gh-pages`, `npm publish`, container/image push, any production rollout).
- **DB writes need explicit user approval** (`INSERT`, `UPDATE`, `DELETE`, schema, migrations). Reads are free. See `db-write-protocol`.
- **Sensitive-data changes need explicit user approval** — hardcoding tokens/keys/secrets in source or test code; storing new sensitive data in `localStorage`/`sessionStorage`/URL; logging auth headers, bearer tokens, cookies, raw PII; weakening route guards, auth guards, or permission checks on either tier. Pause and ask before any such change.
- **Author attribution.** MUST NEVER add `Co-Authored-By: Claude`, `🤖 Generated with [Claude Code]`, "Generated by Anthropic", or any similar AI-attribution trailer/footer to commit messages, PR titles, PR descriptions, issue comments, or release notes. The git author/committer is the human user; assistant identity does not appear in the artifact. This rule overrides any tool default that would inject such a trailer.

### P0.2 Approval-required operations (NEVER run without explicit user `approve` / `yes` / `go ahead`)

| Domain | Operations |
|---|---|
| **Git** | `git push`, `git commit` (incl. `--amend`), `git merge`, `git rebase`, `git tag`, anything to `main`/`master`, any `--force`/`-f`/`--force-with-lease`, `git reset --hard`, `git clean -f`, branch creation (`git checkout -b`, `git switch -c`) |
| **Deploy / publish** | `npm publish`, `yarn publish`, `pnpm publish`, `vercel deploy`, `netlify deploy`, `gh-pages -d`, `gh release create/delete`, `npm version`, container/image push, production rollout |
| **Dependencies** | `npm install --save`/`-S`/`--save-dev`/`-D` (and `yarn add`/`pnpm add`), `npm uninstall`/`npm rm` — any new entry to `dependencies`/`devDependencies`/`peerDependencies`/`optionalDependencies` in any workspace |
| **DB CLIs** | Any `mysql`/`psql`/`mysqldump`/`pgcli`/`mycli`/`sqlite3`/`mongosh`/`prisma migrate` invocation containing `INSERT`, `UPDATE`, `DELETE`, `DROP`, `TRUNCATE`, `ALTER`, `CREATE`, `REPLACE`, `GRANT`, `REVOKE`, `RENAME` |
| **GitHub** | Any `gh` write operation (PR create/merge/close/edit, issue write, release, repo write) |
| **Filesystem** | `rm -rf` (any path) |

### P0.3 Pre-action protocol — for ANY operation in P0.2

1. MUST output the exact command verbatim.
2. MUST output an impact summary scaled to the operation. Per-domain structure: **Git** (`git-workflow`) / **DB** (`db-write-protocol`) / **Deps** (package + bundle/install cost + alternatives + license + maintenance signal) / **Deploy** (environment + blast radius) / **GitHub** (scope + who's notified + state changes).
3. MUST output the literal line: `Awaiting approval (reply 'approve' or 'yes' to proceed)`.
4. MUST stop. MUST NOT execute until the user's next message contains `approve`, `yes`, or `go ahead`.

**Ambiguous replies are NOT approval:** `ok`, `looks fine`, `sounds good`, 👍, silence — re-ask with the exact phrasing required. Forbidden bypass phrases ("I'll just run this", "this is safe", "trivial enough") are NEVER authorization.

## P3 — CODE-CHANGE DEFAULTS

### P3.0 Tier routing — select skills by tier (no cross-tier redundancy)

The frontend and backend live in ONE repo, so the FIRST step of any code task is to identify **which workspace(s) the change touches**, then load ONLY the matching stack skills. Pulling the other tier's stack skills into a single-tier task is redundant context — don't do it.

| Change touches | Load the stack skills for… | Do NOT load |
|---|---|---|
| `apps/web` (frontend) | **Frontend** (Skill Pointers § Frontend) | the Backend skills (`nestjs-*`, `nodejs-best-practices`, `database-transactions`, `db-write-protocol`) |
| `apps/api` (backend) | **Backend** (Skill Pointers § Backend) | the Frontend skills (`react-*`, `accessibility`, `frontend-security`, `vite`/`vitest`/`shadcn`/`tailwind-v4-shadcn`, `bundle-size`, `ai-ui-patterns`, the React design-pattern skills) |
| `packages/contracts` (shared types) | the contract seam (`repo-conventions` § 17) + whichever consuming tier you also edit | — |
| a cross-tier vertical slice (contract + API + UI) | BOTH tier sets + the contract seam | — |

The **Shared / process skills always apply on any tier** — they are stack-neutral and are never "the other tier's skills": `tdd-workflow`, `repo-conventions`, `plan-mode`, `design-review`, `failure-mode-analysis`, `decision-rules`, `bug-investigation`, `rlm-explore`, `code-simplifier`, `cyclomatic-complexity`, `documentation-and-adrs`, `git-workflow`, `pushback-templates`, `meta-skill-hygiene`, `async-error-handling`, `typescript-advanced-types`, `js-performance-patterns`, `cross-repo-workspace`.

The review subagents (P4) apply this SAME routing: each reads only the lens(es) matching the tier(s) the diff touches (Frontend / Backend / Shared-contract), never both stacks for a single-tier diff.

### P3.0.1 Specification-first (before the test)

Before any **behavioral** code change (feature, fix, refactor-that-changes-behavior, follow-up), create/update a Markdown SPEC and resolve material ambiguities **with the user** BEFORE code; reconcile after. Follow `spec-workflow`; `spec-steward` writes it. A cross-tier feature carries one spec per layer (`ui` + `contract`), cross-linked via `counterpart_spec`. Exempt only via `tdd-workflow` waivers; P3.2 applies.

### P3.1 TDD applies to every executable-code change

Use `tdd-workflow`. Failing test first, minimal implementation, run the relevant suite, mini self-review. On a fullstack change, the failing test lives on the tier you're changing (a Vitest/Testing-Library test for `apps/web`, a Jest/Nest test for `apps/api`, a Playwright test for the seam). Either follow TDD or include exactly one of the four valid waiver phrases (`tdd-workflow` § Waivers): `TDD waived — non-code change.` / `TDD waived — type-only.` / `TDD waived — config change with no behavior impact.` / `TDD waived — ADR-only change.`

**Design review applies.** MUST invoke `design-review` before declaring complete, even if its description didn't auto-fire. The response MUST contain a `Design review:` marker plus a `Confidence:` line.

### P3.2 Forbidden non-waivers

"Small change", "obvious fix", "trivial", "just a refactor" are NEVER valid waivers. The check is the rule, not the perceived risk.

### P3.3 High-risk surfaces require requirements restate

If the change touches **auth / sessions / RBAC / payments / secrets / encryption / PII / public-API or shared-contract shape / data-migration / route-guards / `dangerouslySetInnerHTML` / `VITE_*` env vars** — regardless of step or file count — MUST restate the requirements explicitly in your own words BEFORE writing any test or code: what was asked, what's in scope, what's NOT, and any assumptions. This catches misinterpretation early on the surfaces where it's most expensive. The restate happens even if `plan-mode` doesn't fire (e.g. a 1-step change).

### P3.5 Skill-vs-repo conflict resolution

When a skill recommends one thing and `repo-conventions` / this CLAUDE.md says another:

> **Default: follow the skill.** Skills are the curated best-practice catalog and the default source for situational guidance.
>
> **Override:** when applying the skill would require a *structural* change — a new dependency, cross-cutting infrastructure the repo lacks (a global filter, a global `ValidationPipe`, an app-wide store/provider, a request-id middleware, an app-wide bootstrap modification), or a refactor of unrelated modules — **follow the repo convention for the current PR** and recommend the skill's pattern as a Future task in the response's Optional Improvements section.

Test for "structural": would applying this best practice change code outside the current PR's scope? Yes → repo wins, recommend future task. No → skill wins, apply now. The full table with rationale lives in `decision-rules` § 6 (the canonical mirror — keep both in sync; contradiction is a docs bug, flag via `lessons-curator`).

## P4 — MANDATORY VERIFICATION (REVIEW SUBAGENTS)

Subagents run in fresh context to give an independent signal. Each owns ONE concern (no overlap), is willing to BLOCK, and reports a Working Set for RLM attestation. They are **dual-tier**: each applies the frontend lens to `apps/web` changes, the backend lens to `apps/api` changes, and the shared-contract lens to `packages/contracts` changes.

### P4.1 Trigger matrix

| Condition | Subagent |
|---|---|
| Plan with 3+ file changes OR auth/sessions/RBAC/payments/route-guards/state-mgmt-rewrites/data-migration/new-or-changed shared contract | `architect-reviewer` (PRE-impl) |
| Behavioral change (PRE: SPEC + architect review; POST: reconcile spec↔code) | `spec-steward` (PRE + POST) |
| Implementation with 3+ file changes OR auth/sessions/PII/RBAC/payments | `code-reviewer` (POST-impl) |
| Same conditions as `code-reviewer`, run in parallel; also any 1–2 file change that alters observable behavior | `qa-validator` (POST-impl) |
| Auth, sessions, secrets, encryption, payments, PII, RBAC; XSS sinks (`dangerouslySetInnerHTML`, raw HTML), `VITE_*` env vars, postMessage/iframes, file upload/download; SQL/injection surfaces, public-API/contract changes, dependencies | `security-reviewer` (POST-impl) |
| User-facing/API feature OR bug fix that alters observable UI / API / RBAC / multi-step behavior (pure logic/service/util fixes exempt) | `acceptance-verifier` (POST-impl, **after** `qa-validator` green) |
| User correction received | `lessons-curator` (read-only proposer) |

If a trigger fires and you skip the subagent without justification, that's a P8 contract violation.

**`acceptance-verifier` runs LAST** (after `qa-validator` returns green), executes the live suite at the layer that proves each criterion (Playwright e2e for UI flows; Jest integration vs real Postgres + supertest for API/data/RBAC/migration), maps each stated acceptance criterion to an EXECUTED assertion, and adversarially checks non-vacuity (would the green test fail if the feature were reverted?) + surface-fidelity (does it test the surface the spec named?). Verdict **ACCEPTED / GAPS / BLOCK**; its BLOCK is **binding on "done"** (see P8.0).

**Per-PR, not per-session.** Every trigger fires **per pull request**. A session producing N PRs runs the gates N times — a second PR may NOT ride on a first PR's review.

### P4.2 Verdict aggregation

Final confidence = `min(model_rubric, every_subagent_confidence_that_ran)`. **BLOCK** from any subagent → final confidence = 0. Cite the binding subagent when it sets the floor:

> Confidence: 0.85 (set by `qa-validator` coverage gap)

MUST address every HIGH/CRITICAL issue before declaring done. A BLOCK from any reviewer = work is NOT done.

### P4.3 Anti-overlap

`code-reviewer` reviews design only. `qa-validator` reviews coverage / edge cases / a11y / docs only. `security-reviewer` reviews security only. `acceptance-verifier` proves the named acceptance criteria dynamically at the live layer only. `spec-steward` owns `docs/specs/**` only. If you notice a critical gap outside a subagent's mandate, name it briefly and tell the engineer to invoke the appropriate subagent — don't merge concerns.

## P5 — OPERATING MINDSET (ALWAYS-ON DISCIPLINES)

### P5.1 Memory consult — read FIRST

Before any code change, MUST consult the auto-memory system — read the `MEMORY.md` index (loaded at session start) and any linked `feedback`-type memories for the area being changed. Past corrections live there; ignoring them re-creates resolved problems. The `lessons-curator` subagent describes the memory layout and survey procedure.

### P5.2 Scope discipline

Do only the requested task. If adjacent work is valuable, propose it as a follow-up — don't bundle. The cost of an out-of-scope diff is paid by every reviewer.

### P5.3 Surgery over sprawl

Prefer the smallest correct change. A bug fix doesn't need surrounding cleanup; a one-shot operation doesn't need a helper. Three similar lines beat a premature abstraction. Remove only items made obsolete by THIS change; never delete pre-existing dead code unless asked.

### P5.4 Root-cause over symptoms

When something breaks, find why (`bug-investigation`). Patching the symptom (`try { } catch { return null }`) leaves the defect to bite again.

### P5.5 Fail-fast over silent fallback

Validate at boundaries; surface failures with context; never wrap-and-ignore. Don't add retries — the caller (or the server-cache layer's own config) decides retry policy.

### P5.6 Plan before non-trivial work

Use `plan-mode` for 3+ steps, multi-file, architectural, cross-tier, or debugging-with-uncertain-root-cause work. Plans surface assumptions, name failure modes, and prevent scope drift. State assumptions explicitly when they affect behavior, architecture, or delivery risk. Preserve backward compatibility unless the user says otherwise.

### P5.7 Stop and re-plan when evidence contradicts

If a test fails for an unexpected reason, scope expands, or the architecture choice proves weak, STOP and re-plan. Do not push through a flawed plan. When the task IS clear, proceed without hand-holding (bug fixes, CI failures, routine refactors); when ambiguity affects correctness, STOP and ask — never choose silently between interpretations.

### P5.8 Run the relevant suite

Run the full test suite for the tier(s) you touched after every change unless the user explicitly narrows scope. A cross-tier change runs both tiers' suites plus the relevant `e2e/` seam test.

## P6 — DECISION RULES & PUSHBACK

When the request is ambiguous or a default is needed, consult `decision-rules` for the full table. Common defaults:

- **Bug fix scope:** fix only the named bug; surface adjacent issues as follow-ups.
- **Failing test that looks wrong:** the code regressed; ask before changing the test (the asymmetry: silencing a correct test hides a real regression).
- **"Make it faster":** profile first, then propose a targeted change.
- **"Make it cleaner":** surgical, one pass, same scope.
- **Skill description matches but feels wrong for this prompt:** skip it; don't force-fit.
- **Multiple interpretations:** present them numbered; do not pick silently.
- **Ambiguous reply on approval:** treat as NOT approval; re-ask.

When you disagree with the user's framing, see a simpler in-scope alternative, spot scope creep, or detect a hidden risk, push back per `pushback-templates`. State the observation, name the tradeoff, ask the question. One round, then accept the user's call.

## P8 — OUTPUT CONTRACT (FOR CODE CHANGES)

Every code-change response MUST include the following items, in order:

1. **Requirements checklist** — falsifiable bullets per requirement / acceptance criterion.
2. **Working Set / REPL transcript** — only if context is large/dense; cite evidence (file:line).
3. **Plan** — 3–8 steps, each with `verify:` / `files:` / `API impact:` / `tests:` / `risk:` / `slice:`.
4. **Changeset summary** — files + line counts + one line per file, grouped by workspace (`apps/web` / `apps/api` / `packages/contracts` / `e2e`). A reviewer should know what changed in 30 seconds.
5. **Tests** — failing tests written before implementation. Test code appears BEFORE implementation in the response.
6. **Implementation** — minimal; each function traces to a test; no speculative branches. When changing 2+ files, output file-by-file with clear path headers.
7. **How to run / verify** — exact, copy-pasteable commands (the right per-workspace scripts).
8. **Design review block** — principle grid + trade-offs (what was traded off and why) + the calibration rubric below.
9. **Confidence** — rubric outcome, never rounded up.
10. **Optional improvements** — proposals only, no implementation; each with estimated cost + value.

In addition, the response MUST end with a `Skills consulted:` line listing every skill whose `SKILL.md` was read this turn (force-loaded or matched-on-description) — self-attestation; do NOT list skills only seen in the Skill Pointers table.

### P8.0 Definition of "done" (verification artifacts must be EXECUTED)

A change is not "done" — MUST NOT declare it finished, ask the user to test, or open a PR — until:

- **Every verification artifact has been executed**, not merely written. A unit/component/integration/e2e spec that exists but was not run counts as **zero** coverage. An assertion that only checks a serialized shape (e.g. `expect(sql).toContain(...)`) without exercising the real path is **vacuous** and does not count. (Generalizes `tdd-workflow` Step 5 rubric item 2 from the unit level to all layers.)
- **For a user-facing/API feature:** the main agent MUST have authored AND run (a) unit/component/integration tests on the tier(s) changed AND (b) acceptance coverage at the layer that proves the criterion — **Playwright e2e** for UI flows (in `e2e/`), **integration vs real Postgres** or **supertest** for API/data/RBAC/migration — and `acceptance-verifier` MUST have returned non-`BLOCK`. The user's manual testing is then optional, not required.
- **For a bug fix:** a unit/component/integration regression test (authored + run) always; e2e only when the fix changes an observable UI/API/RBAC/multi-step behavior (then `acceptance-verifier` fires).
- A feature with **no stated acceptance criteria** is itself incomplete — write the criteria (in the plan's verification section) before claiming done; "nothing to verify against" is a BLOCK for a user-facing/API feature.
- **Behavioral change:** its SPEC was created/updated, passed the readiness rubric, and `spec-steward` returned non-`BLOCK` (per P3.0.1).

### P8.1 Confidence rubric (5 × 0.20 — strict)

| Item | Points | Earned when |
|---|---|---|
| Tests pass (full suite ran AND green) | 0.20 | The relevant suite(s) ran here and reported green; cite the command. |
| Principles checked (every MUST has a verdict) | 0.20 | Every MUST principle in `design-review` is pass / pass-with-note / fail with a one-line note. |
| No HIGH issues from any reviewer | 0.20 | If any review subagent (`code-reviewer` / `qa-validator` / `security-reviewer` / `architect-reviewer` / `acceptance-verifier` / `spec-steward`) ran, none returned HIGH or BLOCK. |
| Domain gates passed | 0.20 | `security-reviewer` APPROVE for security-touching diffs; `qa-validator` PASS (not GAPS) for coverage-relevant diffs. N/A diffs earn it free. |
| No open assumptions or unresolved questions | 0.20 | Every assumption stated up front is confirmed by code/tests or recorded as a known risk. No "I think"/"should be" hedging on load-bearing facts. |

**Sum < 0.90 → MUST revise the weakest area.** Do not round up. NEVER. The number is the rubric outcome, not a vibe.

### P8.2 Aggregation

`final = min(model_rubric, every_subagent_confidence_that_ran)`. BLOCK from any subagent → `final = 0`. The minimum, not the average. Cite the binding subagent when it sets the floor: `Confidence: 0.85 (set by qa-validator coverage gap)`.

## P9 — STYLE & DEFAULTS

- **Typing.** Strict TypeScript on both tiers. No `any` to silence the compiler — fix the type. Explicit types and clear contracts at boundaries; type the FE↔BE seam from `packages/contracts`.
- **Errors.** Throw with context. Frontend: surface via toast / error boundary / server-cache `error` field; never silent fallback. Backend: centralize error mapping (prefer framework HTTP exceptions); include contextual logging with correlation IDs where infrastructure supports them; degrade gracefully on third-party failure with actionable errors. No retries outside the server-cache library's own config.
- **Logging.** Output redacts tokens / passwords / cookies / raw PII. `frontend-security` (FE) and `repo-conventions` § Logger (BE) enumerate what NEVER goes into logs.
- **Comments.** Default to none. Add only when the WHY is non-obvious; don't restate WHAT well-named identifiers already say.
- **Tests.** Co-located with code (`*.test.tsx` for web, `*.spec.ts` for api). E2E in `e2e/`. Testing-Library queries: role > label > placeholder > text > testId.
- **Response shape.** Concise; state results and decisions directly; running narration on multi-step work, not just a terminal summary. Code references in `path/to/file.ts:NN` form.

## WORKFLOW CHAINS

Task type → skill+subagent recipe. Treat each chain as the default sequence — deviate only with explicit reason.

### Full-stack feature (endpoint + UI + shared contract)

1. `spec-workflow` → `spec-steward` (PRE): author the SPEC(s) per layer + clarification gate, BEFORE code (P3.0.1) — then `plan-mode` (Step 0 + assumptions + slicing across tiers) + `failure-mode-analysis` for the API surface and the UI states
2. If 3+ files OR auth/RBAC/contract-change: `architect-reviewer` (PRE-impl)
3. Define/update the shared type in `packages/contracts` first — it's the seam both tiers build against
4. Backend slice (`apps/api`): `tdd-workflow` → `repo-conventions` + `nestjs-best-practices` / `nestjs-clean-architecture` / `nestjs-patterns` / `database-transactions` as relevant
5. Frontend slice (`apps/web`): `tdd-workflow` → `repo-conventions` + `react-patterns` / `react-state-management` / `react-data-fetching` / `react-routing` / `react-forms` + `accessibility` self-check
6. Seam test in `e2e/` (Playwright) proving FE↔BE agree
7. Run both tiers' suites + the e2e seam → `design-review`
8. POST-impl: `code-reviewer` + `qa-validator`; `security-reviewer` if auth/sessions/PII/XSS-sink/`VITE_*`/SQL/contract/dep; `spec-steward` (POST: reconcile spec↔code); then author + run the e2e/acceptance proving each criterion → `acceptance-verifier` (binding; see P8.0)
9. Output per P8 contract; `Skills consulted:` line at the end

### Frontend feature (route + UI + data)

1. `spec-workflow` → `spec-steward` (PRE) authors the `ui` SPEC first (P3.0.1) → `plan-mode` → `failure-mode-analysis` → (if 3+ files/auth/state-rewrite) `architect-reviewer`
2. `tdd-workflow` per slice → `repo-conventions` + `react-patterns` / `react-state-management` / `react-routing` / `react-data-fetching` / `react-forms`
3. `accessibility` self-check → run frontend suite → `design-review`
4. `code-reviewer` + `qa-validator`; `security-reviewer` if auth/PII/XSS-sink/`VITE_*`/dep; `spec-steward` (POST); then author + run Playwright e2e → `acceptance-verifier` (binding; see P8.0)

### Backend feature / new domain module

1. `spec-workflow` → `spec-steward` (PRE) authors the `contract` SPEC first (P3.0.1) → `plan-mode` (contract-first) → `failure-mode-analysis` → (if 3+ files/auth/migration) `architect-reviewer`
2. `nestjs-clean-architecture` / `nestjs-best-practices` → `tdd-workflow` → `repo-conventions` + `nestjs-patterns` + `database-transactions` (if multi-statement write)
3. run backend suite → `design-review`
4. `code-reviewer` + `qa-validator`; `security-reviewer` if auth/RBAC/SQL/PII; `spec-steward` (POST); then author + run integration/e2e (real Postgres for data/RBAC) → `acceptance-verifier` (binding; see P8.0)

### Bug fix

1. `bug-investigation` (feedback loop, ranked falsifiable hypotheses) — localize to the tier (or the seam); if the fix changes intended behavior, `spec-steward` (PRE) updates the governing SPEC (P3.0.1)
2. `failure-mode-analysis` → `tdd-workflow` (the reproduction is the failing test) → minimal root-cause fix
3. run the relevant suite(s) → `design-review` → `code-reviewer` (3+ files); `qa-validator` for regression coverage; `security-reviewer` if security-adjacent; `spec-steward` (POST) if a SPEC was touched; if the fix alters observable UI/API/RBAC/multi-step behavior, author + run the proving test → `acceptance-verifier` (binding; see P8.0)

### Refactor (no new behavior)

1. `plan-mode` (slicing + verifier per step) → `tdd-workflow` (green at every step) → `code-simplifier` / `cyclomatic-complexity`
2. `repo-conventions` → `design-review` → `code-reviewer` (3+ files); `qa-validator` for coverage parity

### Performance work

1. `rlm-explore` (locate the hot path) → `js-performance-patterns` (+ `react-performance` for rerender cost; `bundle-size` for ship cost)
2. `failure-mode-analysis` → `tdd-workflow` → `repo-conventions` → `design-review` → `code-reviewer` + `qa-validator`

### Async / external-integration code

1. `async-error-handling` → `failure-mode-analysis` (network/partial categories) → `tdd-workflow` → `repo-conventions` → `design-review` → `code-reviewer`

### Structural decision (new persistence layer, auth library, state library, public-API/contract shape)

1. `plan-mode` (with P3.3 restate) → `documentation-and-adrs` (write the ADR alongside) → `tdd-workflow` → `repo-conventions` → `design-review` → `architect-reviewer` + `code-reviewer` + `qa-validator`

### Documentation / ADR-only change

1. If capturing a structural decision: `documentation-and-adrs` for ADR format + citation flow
2. `tdd-workflow` waived — `TDD waived — ADR-only change.` (or `non-code change.` for plain docs); `design-review` waived — non-executable; no subagents required

### User correction received

1. P7 fires immediately: write feedback memory + emit the curator-prompt line
2. Wait for the user to confirm whether to invoke `lessons-curator`
3. If invoked, curator surveys via RLM (no library-wide load) and proposes ONE change; wait for approval before any change lands
